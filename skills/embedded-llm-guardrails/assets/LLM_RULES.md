# LLM 嵌入式开发规则

本文件放在项目 `.claude/` 目录下，用于约束 LLM 在嵌入式、单片机、RTOS、驱动和板级工程中的代码修改行为。具体可改/不可改文件由 `.claude/LLM_BOUNDARY.md` 定义。

---

> **使用说明（LLM 生成时阅读）：**
> 以下内容为模板。`{{...}}` 标记处需要替换为项目实际信息。
> 非标记处为通用规则，直接保留。
> 根据项目类型，启用或删除对应章节。

## 总原则

- 未显式允许修改的文件，默认禁止修改。
- 每次任务必须列出 `Allowed Files`。
- 只做当前任务要求的最小可验证修改。
- 不做顺手重构。
- 不删除、重排、简化或优化已验证代码。
- 修改后必须给出验证方法、硬件风险和下一步确认点。
- 所有实现遵循 TDD 思路：先定义验证，再实现最小代码，再逐步扩展。
- 添加新功能时，必须创建独立的 `.c` 和 `.h` 文件实现功能逻辑，不允许把业务逻辑直接写在 `main.c` 或已有文件中。`main.c` 中只放函数调用。

## 项目边界

项目必须维护 `.claude/LLM_BOUNDARY.md`，至少定义：

- 绝对禁止修改的文件和目录。
- 默认允许修改的文件和目录。
- 需要用户单独确认后才能修改的区域。
- 已验证代码和时序敏感代码位置。
- 主入口和主循环保护点。
- 测试代码和生成代码允许放置位置。

如果没有 `.claude/LLM_BOUNDARY.md` 或没有 `Allowed Files`，LLM 不应修改代码，只能要求用户补充边界。

每次修改前必须复述：

```text
本次允许修改：
本次禁止修改：
本次需要保护的已验证代码：
是否需要用户确认越权修改：
```

## 复杂需求澄清

当需求包含多个功能、多个外设、多个任务、协议栈、状态机、云端通信、OTA、低功耗、生产测试，或用户描述不清时，不要直接写代码。

实现前先输出：

```text
需求理解：
功能拆分：
最小可验证功能：
暂不实现的扩展功能：
需要用户确认的问题：
最小验收标准：
验证方式：
风险点：
```

用户确认前，只能输出需求梳理、接口草案、测试计划、风险点和待确认问题。

## TDD 与最小可验证闭环

所有实现按以下顺序进行：

1. 定义一个最小可验证目标。
2. 定义验证方式（单元测试、主机测试、串口日志、读 ID、固定输入输出、逻辑分析仪波形）。
3. 先写最小测试、最小验证入口或明确的上板验证步骤。
4. 实现刚好能通过该验证的最小代码。
5. 停止扩展，等待验证结果或用户确认。
6. 验证通过后，再进入下一个模块或扩展功能。

如果最小功能可用，但扩展功能失败：

1. 停止继续叠加修改。
2. 回到或保留已验证的最小功能。
3. 记录扩展失败前新增了哪些变量。
4. 重新拆分一个更小的扩展目标。
5. 重新定义验证方法后再实现。

## 默认禁止修改范围

除非任务明确允许，不要修改：

- 启动文件（`startup_*.s`、`startup_*.c`）。
- 链接脚本（`*.ld`、`*.icf`）。
- 时钟树配置（`SystemClock_Config()` 或等效函数）。
- BSP 板级支持包。
- 厂商 SDK / HAL / LL / 标准库。
- 引脚复用和板级初始化。
- ISR 中断服务函数。
- DMA 配置。
- 定时器基础配置。
- RTOS 任务优先级和栈大小。
- 看门狗配置和喂狗点。
- 生产参数、校准参数、设备身份信息。
- OTA、Bootloader、Flash 分区表。
- 已通过硬件验证的时序代码。

## 代码生成器保护（CubeMX / MCUXpresso 等适用）

当项目使用代码生成器时：

- 代码生成器会在生成文件中标记保护区域（如 CubeMX 的 `USER CODE BEGIN xxx` / `USER CODE END xxx`）。
- LLM 只能在保护区域内添加或修改代码。
- 保护区域外的修改会在下次生成时被覆盖，因此禁止修改。
- 如果需要修改保护区域外的代码，必须提醒用户通过代码生成器操作。
- 涉及外设配置变更（新增引脚、修改时钟、启用新外设），应引导用户在代码生成器中操作并重新生成，而不是手动修改生成代码。

## 已验证代码保护

以下标记表示代码已经过硬件验证：

```text
VERIFIED_ON_HARDWARE
DO_NOT_TOUCH_TIMING
LOGIC_ANALYZER_VERIFIED
OSCILLOSCOPE_VERIFIED
PRODUCTION_VERIFIED
```

看到这些标记时：

- 不删除。
- 不重排。
- 不重构。
- 不合并。
- 不优化。
- 不改变延时、顺序、重试次数、GPIO 电平、外设调用顺序。

推荐标记格式：

```c
// VERIFIED_ON_HARDWARE: {{YYYY-MM-DD}}
// DO_NOT_TOUCH_TIMING
// Scope: {{描述受保护的代码范围}}
// Any change requires logic-analyzer or oscilloscope revalidation.
```

## main 和主 while 测试代码

向 `main` 入口、初始化区域或主 `while` 循环添加测试代码时，只允许添加带注释的函数调用。

允许：

- 添加独立测试函数调用。
- 在函数调用上方备注测试目的、临时性和验证方法。
- 将复杂测试逻辑放到独立函数或独立测试文件中。

禁止：

- 在 `main` 或主 `while` 循环中直接写复杂业务逻辑、协议解析、长状态机或大段调试代码。
- 删除已有初始化、调度、通信、喂狗或业务函数调用。
- 为了测试重排已有调用顺序。

临时屏蔽已有调用时，必须备注原因、日期和恢复条件：

```c
// TEMP_DISABLED_FOR_DEBUG: {{YYYY-MM-DD}}
// Reason: {{屏蔽原因}}
// Restore after: {{恢复条件}}
// original_function_call();

// TEMP_TEST: {{测试目的}}
// Remove after board validation passes.
test_function();
```

## 外设驱动开发

编写或改写外设驱动时，不要直接生成完整驱动代码。必须先梳理开发流程，并列出缺失信息，等用户确认后再实现。

必须先确认：

- 目标外设型号、芯片版本、数据手册或寄存器依据。
- MCU 型号、SDK/HAL/LL/裸机或 RTOS 环境。
- 连接方式：I2C/SPI/UART/CAN/GPIO/ADC/PWM、引脚、中断脚、复位脚、电源控制脚。
- 通信参数：地址、片选、电平、速率、模式、字节序、校验方式。
- 初始化流程：上电延时、复位序列、配置寄存器、状态检查。
- API 边界：阻塞/非阻塞、超时、错误码、缓冲区所有权。
- 并发模型：主循环、ISR、DMA、RTOS 任务或回调上下文。
- 最小验证步骤：读 ID、读状态寄存器、回环、固定输出、串口日志、示波器或逻辑分析仪验证。

用户确认前，只能输出：

```text
开发流程：
已知信息：
缺失信息：
需要用户确认的问题：
建议的最小验证步骤：
风险点：
```

## 嵌入式基本准则

资源：

- 默认 RAM、Flash、CPU、栈空间有限。
- 避免无必要的动态内存分配、递归、大栈对象、大范围字符串格式化和浮点运算。
- 新增缓冲区必须说明大小和生命周期。

实时性：

- 不在 ISR 中做耗时操作、阻塞等待、复杂日志或动态分配。
- 不随意增加临界区长度。
- 不随意改变定时器周期、任务周期、任务优先级。

外设：

- GPIO、SPI、I2C、UART、CAN、ADC、PWM、DMA 的顺序可能有硬件含义。
- I2C/SPI/UART/CAN 通信必须考虑超时、错误码、重试和恢复。
- DMA 必须考虑缓存一致性、缓冲区生命周期和并发访问。

并发：

- ISR 与主循环/任务共享的数据必须考虑 `volatile`、原子性、锁、临界区或队列。
- 不无保护地在多个上下文读写同一状态。
- 修改状态机时必须考虑竞态、丢事件、重复事件和异常恢复。

功耗：

- 不随意增加轮询、常驻定时器、常开外设或高频日志。
- 修改低功耗流程时，必须说明唤醒源、外设恢复和功耗影响。

可靠性：

- 通信必须有超时。
- 协议必须有边界检查。
- Flash/EEPROM 写入必须考虑掉电、磨损、校验和版本兼容。
- 看门狗喂狗点不能掩盖死锁。
- 异常路径必须有可观测日志或错误码。

可验证性：

- 编译通过只是最低要求。
- 硬件相关代码必须提供上板验证步骤。
- 时序相关代码必须提供示波器或逻辑分析仪验证建议。
- 协议相关代码必须提供示例报文、golden log 或最小测试入口。
